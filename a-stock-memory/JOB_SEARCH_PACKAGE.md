# AI投研助手 - 求职包装材料

## 简历项目描述

### 50 字版
独立完成并部署 A 股研究辅助应用，围绕重大事件、业绩快照、风险观察和数据质量门禁生成可追溯报告；在免费数据源不稳定时以缓存、状态标识和规则 fallback 保证可用性与合规边界。

### 100 字版
独立完成 React/TypeScript 前端与 FastAPI 后端的 AI 投研助手，部署至 Netlify 与 Render。基于用户访谈将报告重构为重大事件线索、业绩变化概览和风险与后续观察；设计核心 quote/K 线质量门禁、规则 fallback 与 LLM 合规拦截。在线样本中，SH600519 无 Key 规则报告约 1.5 秒返回，且保留 8 个章节、免责声明和数据状态。

### 详细版
AI投研助手是面向个人投资者的公开信息研究辅助项目。产品并不把基础行情“改写成长文”，而是优先解决重大事件、数据限制、业绩快照和风险复核项分散的问题。项目采用 React/TypeScript + FastAPI 构建，前端部署于 Netlify、后端部署于 Render。

我围绕真实用户反馈完成报告信息架构升级：重大事件线索只收录可核验来源，业绩变化概览仅解释最新一期结构化财务事实，风险与后续观察只复用后端事实。为应对免费数据源超时，系统区分核心 quote/K 线与可选增强数据，并以 fresh、stale、mock、fallback 等状态驱动质量门禁。LLM 默认关闭；即使启用，也只能基于事实包整理内容，并受合规与规则 fallback 约束。

## 项目亮点

### 产品亮点
- 由个人投资者访谈确定优先级：打开报告先看重大事件，而非重复基础行情。
- 以“事实、数据限制、待复核事项”组织信息，避免把不完整数据包装成结论。
- 重大事件、业绩和风险模块都明确不是交易建议。

### 技术亮点
- React/TypeScript 前端与 FastAPI/Pydantic 后端分层，报告响应采用向后兼容的可选字段扩展。
- 核心 quote/K 线质量门禁与可选数据降级分离，避免单个非必要数据源阻塞整份报告。
- 单证券行情优先使用既有的单股快照链路，再回退原有全市场路径与缓存。

### AI Agent 协作亮点
- 使用产品、架构、后端、前端、提示词与测试角色分工，按“方案、实现、验收、文档”形成可审计的交付链路。
- 将 Agent 产出限定为服务端事实包、规则和合规边界内，避免自由归因。

### 数据质量与合规亮点
- 显式标注 fresh、stale、mock、fallback、missing 与 partial，不把模拟或降级数据伪装为真实数据。
- 无 Key、LLM 失败、HARD_BLOCK 与最终合规失败都回退到包含免责声明的规则版报告。
- 禁止买卖、仓位、目标价、收益承诺和确定性价格预测。

## 60 秒介绍

我做的是一个面向个人投资者的 AI 投研助手。访谈发现，用户并不缺价格和涨跌幅，真正耗时的是核对重大事件、业绩信息是否完整以及当前数据是否可信。因此我把报告改成三层：重大事件线索、业绩变化概览、风险与后续观察。技术上前端用 React/TypeScript，后端用 FastAPI；quote 和 K 线被作为核心数据做质量门禁，财务、新闻等缺失不会阻塞报告，但会明确显示限制。LLM 默认关闭，即使未来启用也只能整理事实包，任何违规、超时或解析失败都会回退到规则报告。近期 SH600519 的无 Key 线上样本约 1.5 秒返回，并保留 8 个章节和免责声明。

## 3 分钟面试讲解稿

项目起点是一个 A 股行情辅助页面。用户访谈后我发现，把基础行情转成一段文字的信息增量很低，用户更需要先确认近期发生了什么、业绩数据能说明什么、哪些风险和数据限制仍需复核。因此我没有做“自动荐股”，而是把报告设计为信息整理和研究辅助工具。

产品上，报告在元信息之后先展示重大事件线索，再展示业绩变化概览和风险观察。重大事件只有来源、日期和状态可核验时才进入列表；普通新闻不会被包装成正式公告。业绩模块只使用最新一期的营收增速、净利润增速、毛利率、ROE、资产负债率和 EPS，明确不做完整原因归因。风险模块复用服务端已有事实，并给出证据和复核来源。

技术上，前端使用 React、TypeScript、Vite，后端使用 FastAPI、Pydantic。报告接口维持 8 个固定章节，并通过可选顶层字段兼容新增模块。免费数据源不稳定是最大工程难点：我把 quote 和 K 线定义为核心数据，mock、fallback、严重 stale 或明显不一致时不允许显示成正常 AI 报告；detail、新闻和财务属于可选增强数据，缺失时继续生成但明确标注。针对慢请求，研究服务采用有界聚合和缓存优先策略；单证券行情也增加了既有单股快照优先路径。近期线上 SH600519 无 Key 报告样本约 1.5 秒返回。

AI 的边界同样是设计重点。LLM 默认关闭；如果在私有后端环境启用，它只能基于服务端事实包整理内容。交易动作、仓位、目标价、收益承诺与确定性预测由合规规则阻断；失败则返回规则 fallback，不暴露密钥或原始模型内容。项目仍有财务和资金流上游不稳定等限制，但我选择透明降级而不是伪造实时数据。

## 高频面试问题

### 为什么做这个项目？
用户访谈表明基础行情已高度同质化，难点在于把事件、数据限制和复核事项快速组织起来。我选择做研究辅助而不是荐股。

### 用户需求来自哪里？
来自有实际炒股经验的个人投资者访谈。核心反馈是第一眼想看到重大事件，并理解业绩、风险和数据是否可信。

### 为什么不用纯 LLM？
纯 LLM 容易把缺失数据补成看似合理的故事。项目先由服务端规则构造事实、状态和限制，LLM 只能重写已有事实，失败时可安全回退。

### 数据源超时如何处理？
区分核心数据与可选数据：核心 quote/K 线影响质量门禁；可选数据超时后报告继续生成并显示缺失。缓存、状态标识和有界等待避免把单个慢源变成全链路阻塞。

### 如何防止 AI 编造？
事实包白名单、固定 schema、prompt 约束、合规扫描、HARD_BLOCK 和规则 fallback 共同限制；模型不直接决定事件、财务原因或风险等级。

### 如何保证合规？
产品明确不提供买卖、仓位、目标价、收益承诺或确定性预测；所有报告包含固定免责声明，并对可展示文本执行合规检查。

### 最大技术难点是什么？
免费数据源的可用性和线程生命周期。外层 future timeout 不能停止底层阻塞调用，因此改为研究报告可选数据缓存/fallback 优先，并为核心数据设置质量门禁。

### 项目还存在哪些不足？
当前财务数据常为空，资金流上游可能中断；业绩模块只能做单期快照，重大事件模块是线索而非正式公告中心。

### 下一步如何扩展？
先评审稳定且合规的数据源，再考虑正式公告、估值、行业因素、多期趋势和报告历史；不会扩展自动荐股或交易指令。

### 个人实际负责了什么？
请按个人真实经历表述。推荐表述：我负责从用户访谈需求拆解、前后端实现、质量门禁与 fallback 设计、部署验证到文档和演示材料整理；涉及的协作角色用于分工与复核，不替代本人对最终实现的负责。
